# 项目架构

## 概述

tree-sitter-gdscript 是 GDScript（Godot 游戏引擎内置脚本语言）的 tree-sitter 语法解析器，版本 6.1.0。

- 原作者：Preston Knopp
- 许可：MIT
- 同步的 Godot 提交：[6ae54fd787](https://github.com/godotengine/godot/commits/6ae54fd787)
- 当前分支用于个人定制改进（region 折叠、not/! 优先级修复等），配合 nvim-treesitter 使用

## 目录结构

```
tree-sitter-gdscript/
├── grammar.js           # 核心语法定义（934 行），唯一需要手动维护的语法文件
├── src/
│   ├── parser.c         # 由 grammar.js 自动生成的解析器（~97000 行），不要手动编辑
│   ├── scanner.c        # 手写外部扫描器（629 行），处理缩进、字符串、行续行
│   ├── grammar.json     # 自动生成的语法 JSON 表示
│   ├── node-types.json  # 自动生成的节点类型注册表
│   └── tree_sitter/     # tree-sitter 运行时头文件
├── scm/
│   ├── highlights.scm   # 语法高亮规则（636 行）
│   ├── folds.scm        # 代码折叠规则（28 行）
│   ├── indents.scm      # 缩进规则（79 行）
│   ├── locals.scm       # 局部变量作用域规则（122 行）
│   └── injections.scm   # 语言注入规则（2 行）
├── test/corpus/         # 测试用例（17 个文件，4606 行）

├── bindings/            # 多语言绑定（c, go, node, python, rust, swift, nim）
├── build/               # 编译产物
├── prebuilds/           # 预编译二进制文件
├── tree-sitter.json     # 项目元数据（名称、scope、文件类型）
├── package.json         # Node.js 包配置
├── Cargo.toml           # Rust crate 配置
├── pyproject.toml       # Python 包配置
├── binding.gyp          # node-gyp 构建配置

└── Makefile             # C 库构建
```

## 核心编译流程

```
grammar.js  ──tree-sitter generate──>  src/parser.c
                  │                      src/grammar.json
                  │                      src/node-types.json
                  v
src/scanner.c ──(手写，不参与生成)──┐
                                    ├── 编译 ──> parser 动态库 (.so/.dylib/.node)
src/parser.c ──────────────────────┘
```

关键命令：

| 命令 | 作用 |
|------|------|
| `tree-sitter generate` | 从 grammar.js 生成 parser.c |
| `tree-sitter test` | 运行 test/corpus/ 中的测试 |
| `npm run genTest` | generate + test（开发主循环） |
| `npm run test` | 仅测试（scanner.c 改动后用这个） |
| `npm run prebuild` | 构建预编译二进制文件 |

## tree-sitter 基本概念

### Grammar DSL

grammar.js 使用 JavaScript DSL 定义语法：

- `seq(a, b)` — 序列，a 后跟 b
- `choice(a, b, ...)` — 选择，匹配其中一个
- `repeat(rule)` — 零或多次重复
- `repeat1(rule)` — 一或多次重复
- `prec(n, rule)` — 设置优先级 n
- `prec.left(n, rule)` — 左结合优先级
- `prec.right(n, rule)` — 右结合优先级
- `field(name, rule)` — 给节点添加字段名
- `token(rule)` — 在词法层面匹配（不产生子节点）
- `alias(rule, name)` — 给规则取别名
- `optional(rule)` — 可选匹配

### External Scanner

grammar.js 中 `externals` 声明的 token 由手写 C 代码（scanner.c）处理。这些 token 涉及上下文状态（缩进栈、字符串分隔符栈），无法用纯 grammar DSL 表达。

### Query 文件

SCM 文件使用 S-expression 模式匹配 AST 节点，供编辑器实现：

- **highlights.scm** — 语法高亮捕获组（如 `@keyword`, `@function`, `@type`）
- **folds.scm** — 定义哪些节点可折叠
- **indents.scm** — 定义缩进规则
- **locals.scm** — 定义作用域和变量定义/引用关系
- **injections.scm** — 在宿主语言中嵌入其他语言

## 文件依赖关系

```
grammar.js
  ├── 定义 externals → scanner.c 必须处理这些 token
  ├── 定义规则名 → test/corpus/*.txt 引用这些规则名
  ├── 定义 supertypes → scm/*.scm 模式匹配使用这些类型
  └── 生成 → parser.c, node-types.json

scanner.c
  └── 处理 externals 中的 14 个 token

scm/*.scm
  ├── 依赖 grammar.js 的规则名和节点结构
  └── 被 nvim-treesitter 等编辑器插件消费

test/corpus/*.txt
  └── 引用 grammar.js 中的规则名，验证 AST 输出
```

**修改 grammar.js 时需同步更新**：

1. `test/corpus/` — 添加/修改测试用例
2. `scm/` — 如果改了规则名或节点结构
3. `src/scanner.c` — 如果改了 externals 列表
4. 运行 `npm run genTest` 重新生成并测试

## Neovim 集成

使用方式（个人定制流程）：

1. 在 nvim-treesitter 仓库修改 `lua/nvim-treesitter/parsers.lua`，将 gdscript 指向本仓库和最新提交
2. 启动 nvim，执行 `:TSInstall gdscript`
3. 将查询目录指向本仓库 `./scm`：修改 `~/.local/share/nvim/site/queries/gdscript` 软链接
