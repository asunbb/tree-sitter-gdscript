# 构建与开发

---

## 前置准备

- Node.js v20-24
- tree-sitter CLI：`npm install -g tree-sitter-cli`（或使用 npx）
- C 编译器（用于 node-gyp 编译原生模块）
- 可选：Cargo（Rust 绑定）、Python 3.9+（Python 绑定）

---

## npm scripts 参考

| 命令 | 作用 |
|------|------|
| `npm run generate` | 从 grammar.js 生成 parser.c（别名 `npm run gen`） |
| `npm run test` | 运行 test/corpus/ 中的测试（不重新生成） |
| `npm run genTest` | generate + test（开发主循环） |
| `npm run parse` | 解析指定文件并输出 AST（`tree-sitter parse <file>`） |
| `npm run prebuild` | 构建预编译二进制文件（Node 20-24, Electron 34-36） |
| `npm run start` | 启动浏览器端 tree-sitter playground |
| `npm run format` | 用 prettier 格式化 grammar.js 和 package.json |
| `npm run versions` | 显示 node/npm/tree-sitter 版本 |

## 开发循环

### 步骤 1：编写测试用例

在 `test/corpus/` 中编写或修改测试文件。

### 步骤 2：修改代码

- **修改语法规则**：编辑 `grammar.js`
- **修改外部扫描器**：编辑 `src/scanner.c`
  - scanner.c 的修改**不需要** `generate`，只需 `test`

### 步骤 3：运行测试

```bash
# grammar.js 有改动
npm run genTest

# 仅 scanner.c 有改动
npm run test
```

### 步骤 4：格式化并提交

```bash
npm run format
git add ...
git commit
```

## 测试用例格式

测试文件使用以下格式：

```
测试名称
===

输入代码
---

期望的 AST
```

### 示例

```
Variable declaration
===

var x = 10

---
(source
  (variable_statement
    (name)
    (integer)))
```

### 多个测试

一个文件可包含多个测试，每个以 `===` 和 `---` 分隔：

```
Test 1
===

input1

---
(expected_ast_1)

Test 2
===

input2

---
(expected_ast_2)
```

## 部署到 Neovim

完整工作流程：

1. **修改 parsers.lua**

   在 nvim-treesitter 仓库中编辑 `lua/nvim-treesitter/parsers.lua`：

   ```lua
   local gdscript = {
     install_info = {
       url = "/path/to/tree-sitter-gdscript",
       revision = "你的最新提交 hash",
     }
   }
   ```

2. **安装解析器**

   在 Neovim 中执行：

   ```
   :TSInstall gdscript
   ```

   nvim-treesitter 会：
   - 下载本仓库并编译 `gdscript.so`
   - 放置到 `~/.local/share/nvim/site/parser/`

3. **设置查询目录**

   将查询目录软链接到本仓库的 `./scm`：

   ```bash
   ln -sf /path/to/tree-sitter-gdscript/scm ~/.local/share/nvim/site/queries/gdscript
   ```

4. **验证**

   在 Neovim 中打开 `.gd` 文件，检查高亮、折叠、缩进是否正常。

## 自定义修改记录

### 1. Region 折叠支持

**修改内容**：

- `grammar.js`：新增 `region_start`、`region_end`、`region` 三条规则
- `grammar.js`：`_compound_statement` 添加 `$.region`
- `grammar.js`：`_compound_class_member` 添加 `$.region`
- `grammar.js`：`match_body` 内部支持 `region_start`/`region_end`
- `src/scanner.c`：注释处理中添加 region 标记检测，不调整缩进
- `scm/folds.scm`：添加 `(region)` 到折叠列表
- `test/corpus/regions.txt`：新增 183 行测试

**解决的问题**：之前的 `#region`/`#endregion` 被当作普通注释解析，无法折叠。且在类体和 match 体中，region 标记会干扰缩进解析。

### 2. not/! 优先级修复

**修改内容**：

- `grammar.js`：`not_operator` 使用 `PREC.boolean_not`(5) 替代原来的 `PREC.unary`(15)
- `test/corpus/not_precedence.txt`：新增 311 行测试

**解决的问题**：`not a == b` 被解析为 `(not a) == b` 而非预期的 `not (a == b)`。将优先级从 15 降至 5 后，not 绑定松于比较运算符(7)但紧于 and(4)。

### 3. 内部类 region 修复

**修改内容**：

- `grammar.js`：`_compound_class_member` 添加 `$.region`

**解决的问题**：之前 `#region`/`#endregion` 在类体中无法正确解析，因为 `_compound_class_member` 不包含 region。

## 版本发布流程

（参考，来自 README）

1. 在所有包文件中更新版本号：`CMakeLists.txt`, `Cargo.toml`, `Makefile`, `pyproject.toml`, `tree-sitter.json`
2. `npm version --git-tag-version false <major|minor|patch>`
3. `git tag -a v<version>`
4. `git push && git push --tags`
5. `cargo package && cargo publish`
