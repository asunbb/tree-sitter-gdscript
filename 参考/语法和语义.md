# 语法高亮 vs 语义高亮

## 疑问

类的属性在类成员函数中使用时，没有按照 property 高亮着色。

## 原因

这是 tree-sitter 的固有局限 — 它只能做**语法高亮**，无法做语义分析。

当前属性高亮规则只匹配声明处：

```scheme
(class_definition
  (class_body
    (variable_statement
      (name) @property)))
```

示例：

```gdscript
class MyClass:
    var my_prop = 10        # 声明 → 匹配 → @property

    func my_method():
        print(my_prop)       # 使用 → 只是 (identifier) → @variable
```

`my_prop` 在方法体内使用时是一个普通的 `(identifier)` 节点，tree-sitter 无法区分它引用的是类属性还是局部变量 — 这需要语义分析，超出了 tree-sitter 的能力范围。

## 可行的替代方案

通过 `self.` 访问时可以正确高亮：

```gdscript
func my_method():
    print(self.my_prop)      # my_prop → @property
```

因为已有规则匹配 `attribute` 节点中的 `identifier` 为 `@property`：

```scheme
(attribute
  (_)
  (identifier) @property)
```

## 疑问：类静态函数调用如何使用特殊高亮组

### 问题

`MyClass.static_func()` 和 `obj.method()` 解析为相同的 AST 结构，无法区分静态调用与实例方法调用。

### AST 结构

`MyClass.static_func()` 解析为：

```
call
  attribute
    identifier "MyClass"
    "."
    identifier "static_func"
  arguments "(...)"
```

### 语法层面的局限

`grammar.js` 中没有专门标注静态调用的 AST 节点。`static_keyword` 仅出现在：
- 变量声明中的 `static var`
- 函数定义中的 `static func`

调用处没有静态标注，只能通过命名约定（类名以大写字母开头）判断。

### 高亮方案

利用 `#lua-match?` 检查 attribute 基部是否以大写字母开头：

```scheme
(call
  (attribute
    (identifier) @type
    "."
    (identifier) @function.call)
  (#lua-match? @type "^[A-Z]"))
```

- 静态方法名 → `@function.call`（普通函数调用）
- 实例方法名 → 保持 `@function.method.call`
