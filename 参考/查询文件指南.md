# 查询文件指南 (SCM)

`scm/` 目录包含 5 个查询文件，使用 S-expression 模式匹配 AST 节点，供编辑器（如 Neovim）实现语法高亮、代码折叠、缩进、作用域分析和语言注入。

---

## tree-sitter 查询语法基础

### 基本模式

```scheme
(node_name) @capture              # 匹配节点，命名为 @capture
(node_name (child))               # 匹配含特定子节点的节点
(node_name field: (child))        # 通过字段名匹配
(node_name (_) (child))           # _ 匹配任意子节点
```

### 捕获组

以 `@` 开头的名称（如 `@keyword`, `@function.call`）。Neovim-treesitter 定义了标准的捕获组名，映射到相应的高亮组。

### 谓词

```scheme
(#lua-match? @capture "pattern")   # Lua 正则匹配捕获内容
(#eq? @capture "value")            # 精确等于
(#any-of? @capture "a" "b" ...)   # 匹配列表中的任意值
(#set! key value)                  # 设置元数据
```

### 查询顺序

查询从上到下匹配，**后面的匹配会覆盖前面的**。因此：
- 通用规则在前
- 特定规则在后（利用覆盖机制实现更精确的匹配）

## highlights.scm (636 行)

最大且最复杂的查询文件，定义了所有语法高亮规则。

### 基本高亮

| 规则 | 捕获组 | 说明 |
|------|--------|------|
| `(identifier)` | `@variable` | 所有标识符默认为变量 |
| `(name)` | `@variable` | 命名符号默认为变量 |
| `(type (identifier))` | `@type` | 类型标注中的标识符 |
| `(comment)` | `@comment @spell` | 注释（启用拼写检查） |
| `(string)` | `@string` | 字符串 |
| `(string_name)` | `@string` | 字符串名 |
| `(float)` | `@number.float` | 浮点数 |
| `(integer)` | `@number` | 整数 |
| `(null)` | `@constant` | null |
| `(true)` `(false)` | `@boolean` | 布尔值 |
| `(get_node)` `(node_path)` | `@string.special.url` | 节点路径 |

### 类与属性

```scheme
; 脚本文件级 class_name 声明
(class_name_statement (name) @type) @keyword

; 类定义中的类名
(class_definition (name) @type)

; 类体中的变量 → @property（Godot 术语）
(class_definition (class_body (variable_statement (name) @property)))

; 类体中的函数 → @function.method
(class_definition (class_body (function_definition (name) @function.method)))
```

**注意**：属性高亮只在声明处生效，使用时不会高亮——这是 tree-sitter 的语法高亮局限（详见 `语法和语义.md`）。

### 常量

```scheme
(const_statement "const" @keyword.modifier (name) @constant)
```

### 函数

```scheme
; 函数定义名
(function_definition (name) @function)

; 构造函数
(constructor_definition "_init" @constructor)

; 函数调用
(call (identifier) @function.call)

; load/preload 特殊标记
(call (identifier) @keyword.import (#any-of? @keyword.import "preload" "load"))

; 方法调用
(attribute_call (identifier) @function.method.call)

; .new() 特殊标记
(attribute_call (identifier) @keyword.operator (#eq? @keyword.operator "new"))
```

### 参数

所有四种参数类型的标识符都标记为 `@variable.parameter`：

```scheme
(parameters (identifier) @variable.parameter)
(typed_parameter (identifier) @variable.parameter)
(default_parameter (identifier) @variable.parameter)
(typed_default_parameter (identifier) @variable.parameter)
```

### 属性访问

```scheme
; a.b 中的 b → @property
(attribute (_) (identifier) @property)

; 下标中的属性名
(attribute_subscript (identifier) @property)
```

### 命名约定

```scheme
; 大写开头 → @type（类名、类型名）
((identifier) @type (#lua-match? @type "^[A-Z]"))

; 全大写+下划线 → @constant（枚举值、常量名）
((identifier) @constant (#lua-match? @constant "^[A-Z][A-Z_0-9]*$"))
```

**重要**：命名约定查询放在属性查询之后，确保 `type (attribute (identifier))` 的匹配不会覆盖属性高亮。

### 枚举

```scheme
(enumerator left: (identifier) @constant)
```

### 内建

#### 内建变量

```scheme
((identifier) @variable.builtin (#any-of? @variable.builtin "self" "super"))
```

#### 内建注解

```scheme
((annotation "@" @attribute (identifier) @attribute)
  (#any-of? @attribute "abstract" "export" "export_category" ...))
```

列出了约 30 个 Godot 内建注解（从 `@GDScript.xml` 提取）。

#### 内建类型

通过 `#any-of?` 列出了约 500 个 Godot 内建类型（从 `doc/classes/*.xml` 提取），标记为 `@type.builtin`。

#### 内建函数

列出约 150 个全局函数（从 `@GlobalScope.xml` 和 `@GDScript.xml` 提取），标记为 `@function.builtin`。

#### 内建常量

列出约 600 个常量（`INF`, `NAN`, `PI`, `TAU`, `KEY_*`, `TYPE_*`, `OP_*` 等），标记为 `@constant.builtin`。

### 关键字与运算符

| 类别 | 关键字 | 捕获组 |
|------|--------|--------|
| 条件 | `if` `elif` `else` `match` | `@keyword.conditional` |
| 循环 | `for` `while` `break` `continue` | `@keyword.repeat` |
| 返回 | `return` | `@keyword.return` |
| 函数定义 | `func` | `@keyword.function` |
| 协程 | `await` `yield` | `@keyword.coroutine` |
| 类型 | `enum` `class` | `@keyword.type` |
| 导入 | `export` | `@keyword.import` |
| 修饰符 | `static` `const` | `@keyword.modifier` |
| 通用关键字 | `pass` `class_name` `extends` `signal` `var` `onready` 等 | `@keyword` |
| 运算符关键字 | `and` `as` `in` `is` `not` `or` | `@keyword.operator` |
| 符号运算符 | `+` `-` `*` `/` `==` `!=` `>` `<` 等 | `@operator` |
| 标点 | `( ) [ ] { }` | `@punctuation.bracket` |
| 标点 | `,` `.` `:` `;` | `@punctuation.delimiter` |

### 特殊标记

```scheme
; match pattern 开放结尾
(pattern_open_ending) @character.special

; 下划线变量（未使用变量）
((identifier) @character.special (#eq? @character.special "_"))

; doc string（表达式语句中的字符串）
(expression_statement (string) @comment @spell)

; setter/getter 函数名
(setter) @function
(getter) @function
(set_body "set" @keyword.function)
(get_body "get" @keyword.function)
```

## folds.scm (28 行)

定义可折叠的代码块：

```scheme
[
  (if_statement) (elif_clause) (else_clause)
  (for_statement) (while_statement)
  (class_definition) (enum_definition)
  (match_statement) (pattern_section)
  (function_definition) (lambda) (constructor_definition)
  (region)   ← 自定义新增
] @fold
```

### `(body)` 折叠被注释掉的原因

```scheme
; Body fold will "join" the next adjacent fold into a SUPER fold.
; This is an issue with the grammar.
; (body)
```

如果启用 `(body)` 折叠，它会"吞噬"相邻的折叠区域，导致 if/elif/else 等无法单独折叠。

### if 语句的额外折叠点

```scheme
(if_statement (body) @fold)
```

让 if 语句的 body 也可以单独折叠（除了折叠整个 if 之外）。

### 文档字符串折叠

```scheme
(expression_statement (string) @fold)
```

函数/类开头的字符串（doc string）可折叠。

### region 折叠

`(region)` 被添加到折叠列表中（自定义新增），使 `#region`/`#endregion` 之间的代码可以折叠。

## indents.scm (79 行)

### 启用缩进的节点

```scheme
[
  (lambda) (function_definition) (constructor_definition)
  (for_statement) (while_statement) (if_statement)
  (class_definition) (match_statement) (pattern_section)
  (setget) (match_body) (set_body) (get_body)
] @indent.begin
```

### 分支缩进

```scheme
[(elif_clause) (else_clause)] @indent.branch
```

### 自动缩进（括号匹配）

```scheme
[(string) (comment) (array) (dictionary) (parenthesized_expression) (ERROR)] @indent.auto
```

### 去缩进关键字

```scheme
[(pass_statement) (continue_statement) (break_statement) (return_statement)] @indent.dedent
```

### 已知问题

文件中大量注释掉的规则说明：

- `(parameters)` 和 `(arguments)` 的对齐缩进（`@indent.align`）效果不佳
- 解析器贪婪消费空白字符，导致在 body 最后一行交互式编辑时缩进行为不理想
- 当前使用 `@indent.auto` 作为折中方案
- `(arguments)` 作为 `@indent.begin` 使用 tabs 时工作正常，但用 spaces 时有问题

```scheme
; These only work with expanded tabs.
; ((parameters) @indent.align ...)
; ((arguments)  @indent.align ...)
```

## locals.scm (122 行)

定义作用域和变量定义/引用关系，支持编辑器的"跳转到定义"和"查找引用"功能。

### 作用域

```scheme
[
  (if_statement) (elif_clause) (else_clause)
  (for_statement) (while_statement)
  (function_definition) (constructor_definition)
  (class_definition) (match_statement) (pattern_section)
  (lambda) (get_body) (set_body)
] @local.scope
```

### 参数定义

```scheme
(parameters (identifier) @local.definition.parameter)
(default_parameter (identifier) @local.definition.parameter)
(typed_parameter (identifier) @local.definition.parameter)
(typed_default_parameter (identifier) @local.definition.parameter)
```

### 变量定义

| 上下文 | 捕获组 |
|--------|--------|
| `const_statement` 的 name | `@local.definition.constant` |
| `variable_statement` 的 name | `@local.definition.var` |
| 脚本级 `variable_statement` | `@local.definition.field` |
| 脚本级 `onready_variable_statement` | `@local.definition.field` |
| 脚本级 `export_variable_statement` | `@local.definition.field` |
| 类体中的 `variable_statement` | `@local.definition.field` |
| `for` 语句的迭代变量 | `@local.definition.var` |
| `pattern_binding` 的变量 | `@local.definition.var` |

### 函数/方法定义

```scheme
; 函数定义，scope = "parent" 使其在父作用域可见
((function_definition (name) @local.definition.function)
  (#set! definition.function.scope "parent"))

; Lambda 名称仅用于调试，不可访问
(lambda (name) @local.definition.function)

; 类体中的方法，scope = "parent"
((class_definition (class_body (function_definition (name) @local.definition.method)))
  (#set! definition.method.scope "parent"))
```

`scope "parent"` 使定义在声明它的作用域的**父作用域**中可见。这让函数名可以在定义之后被引用。

### 类型和枚举

```scheme
(class_name_statement (name) @local.definition.type)

((class_definition (name) @local.definition.type)
  (#set! definition.type.scope "parent"))

(enum_definition (name) @local.definition.enum)
```

### Signal

```scheme
(signal_statement (name) @local.definition.field)
```

Signal 被定义为 field（属性），因为 GDScript 2 中 signal 的行为类似属性。

### 引用

```scheme
(identifier) @local.reference
(setter) @local.reference
(getter) @local.reference
```

所有标识符都是引用。setter/getter 名称也是引用（指向对应的函数）。

### 脚本 vs 类体的区别

```
source (脚本顶层):
  variable_statement   → @local.definition.field（脚本本身就是类）
  onready_variable     → @local.definition.field
  export_variable      → @local.definition.field

class_definition (内部类):
  variable_statement   → @local.definition.field
  function_definition  → @local.definition.method
```

## injections.scm (2 行)

```scheme
((comment) @injection.content
  (#set! injection.language "comment"))
```

将注释内容注入为 "comment" 语言，启用注释内的拼写检查等子语言功能。

## 修改查询文件的实用建议

### 测试查询修改

1. 修改 `scm/` 下的查询文件
2. 确保查询目录指向本仓库：`~/.local/share/nvim/site/queries/gdscript` → `./scm`
3. 在 Neovim 中重新打开文件，观察效果

### 调试工具

- `:TSPlaygroundToggle` — 打开 tree-sitter playground，查看 AST 和高亮捕获
- `:edit` — 重新加载文件以应用查询更改

### 添加新高亮规则

1. 在 highlights.scm 中找到合适的类别位置
2. 注意查询顺序：特定规则应放在通用规则之后
3. 使用 Neovim 标准捕获组名（参见 nvim-treesitter 文档）

### 语法变更后的查询同步

修改 grammar.js 规则名或节点结构后，需检查：

1. highlights.scm — 是否引用了被改名的规则
2. folds.scm — 新的复合语句是否需要添加到折叠列表
3. indents.scm — 新的块结构是否需要缩进规则
4. locals.scm — 新的定义位置是否需要添加作用域规则
